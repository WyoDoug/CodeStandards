# ENC0003: Public/protected fields

| Property | Value |
|----------|-------|
| **ID** | ENC0003 |
| **Category** | Encapsulation |
| **Severity** | Warning |
| **Has Code Fix** | Yes |

## Description

Public and protected fields break encapsulation. Fields should be private; expose state through properties instead.

## Non-compliant Code

```csharp
public class Person
{
    public string Name; // ENC0003
    public int Age; // ENC0003
    protected double mSalary; // ENC0003
}
```

## Compliant Code

```csharp
public class Person
{
    public string Name { get; set; }
    public int Age { get; set; }

    private double mSalary;
    protected double Salary
    {
        get => mSalary;
        set => mSalary = value;
    }
}
```

## Exemptions

- **Structs** — Value types commonly use public fields for performance and interop
- **Records** — Use compiler-generated properties by design
- **Types with `[StructLayout]` attribute** — Interop scenarios requiring specific memory layout
- **WPF types** — Classes deriving from `DependencyObject`, `FrameworkElement`, etc.
- **Readonly/const fields** — Immutable fields are safe to expose
- **Static fields** — Class-level state (different encapsulation concern)

## When to Suppress

Suppress for legacy serialization compatibility or interop requirements.

```csharp
#pragma warning disable ENC0003 // Public field required for legacy serialization compatibility
public string LegacyData;
#pragma warning restore ENC0003
```
